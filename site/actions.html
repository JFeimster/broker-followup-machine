Actions planning

Keep integrations separated by auth type.

The Action plan should make it easy to see what can ship without auth, what needs an API key, and what belongs behind OAuth and explicit review.

None

Start without an Action

  • Pure documentation workflows
  • Template-only use cases
  • Static examples and uploadable files

API Key

Key-based categories

  • Simple read or write operations with a shared key
  • Low-complexity utility calls
  • Systems where the operator controls the key handling

OAuth

Consent-based categories

  • Connected CRM or workspace actions
  • User-specific access to files or records
  • Operations that should be tied to a signed-in user

No-auth

Public categories

  • Static documentation retrieval
  • Public examples and templates
  • Safe browse-only content that does not expose private data

Safety boundaries

Where Action use should stop and review should start

  • Require review before borrower-facing claims
  • Require review before lender-specific statements
  • Require review before decline messages or underwriting-related notes
  • Require review before handling sensitive documents or personal data
  • Keep auth separation clear so each action path is easy to audit
